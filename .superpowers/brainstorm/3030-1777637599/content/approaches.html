Как технически определять метод входа?

Три подхода — выбери который нравится

📧 Пользователь вводит email
↓ запрос к /api/auth/check
🔍 Сервер проверяет Supabase Admin API
✅ Есть пароль
→ поле пароля
📨 Нет пароля
→ OTP код
Новый пользователь → OTP

A — Проверка через Admin API ⭐

Сервер проверяет Supabase и точно знает есть ли у пользователя пароль. Чистый UX, без догадок.

Плюсы

  • Точно и надёжно
  • Лучший UX

Минусы

  • Один доп. API-запрос
📧 Email + попытка signInWithPassword
✅ Успех
→ залогинен
❌ Ошибка
→ показать OTP
⚠️ Пользователю виден «неверный пароль»

B — Попытка и fallback

Сначала пробуем войти с паролем, если ошибка — переключаемся на OTP. Проще в коде, но запутывает пользователя.

Плюсы

  • Проще реализация

Минусы

  • Плохой UX при ошибке
  • Непонятные сообщения
📧 OTP вход (как сейчас)
↓ после входа
👤 В профиле: «Установить пароль»
↓ следующий раз
🔑 Email + пароль (как в варианте A)

C — Пароль устанавливается в профиле

Первый вход — всегда OTP. Потом в профиле пользователь сам устанавливает пароль. Следующие входы — через пароль.

Плюсы

  • Постепенный переход
  • Не ломает текущий UX

Минусы

  • 2 шага для настройки
